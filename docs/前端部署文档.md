# 前端部署到 Railway - 超详细手把手教程

## 准备工作

1. 一个 GitHub 账号
2. 你的代码已经推送到 GitHub 上的 `ai-lowcode-platform` 仓库
3. 一个 Railway 账号（用 GitHub 登录就行）
4. **服务端已经部署好了，并且有了服务端的地址**（比如：`https://xxx.railway.app`）

---

## 第一步：先添加必要的配置文件（重要！）

在部署之前，我们需要先给前端添加两个配置文件，你应该已经在项目里了：

1. `web/Dockerfile` - Docker 构建文件
2. `web/nginx.conf` - nginx 配置文件（解决路由刷新 404 问题）

如果没有，请确保这两个文件在 `web/` 目录下，然后提交并推送到 GitHub：

```bash
git add web/Dockerfile web/nginx.conf
git commit -m "add Dockerfile and nginx config for frontend"
git push
```

---

## 第二步：开始部署（每一步都跟着做）

### 第 1 步：打开 Railway 并登录

1. 打开浏览器，访问：https://railway.app
2. 点击右上角的 **Login** 按钮
3. 选择 **Sign in with GitHub**
4. 授权 Railway 访问你的 GitHub 账号（点同意/Authorize）
5. 登录成功后，你会看到 Railway 的控制台（Dashboard）

---

### 第 2 步：在同一个项目里添加前端服务

1. 在你之前创建的项目里（就是有后端的那个项目），点击 **New** → **Service**
2. 选择 **GitHub repo**
3. 找到 `ai-lowcode-platform` 仓库，点击旁边的部署按钮

---

### 第 3 步：配置前端服务（关键！）

在服务配置页面，仔细配置：

1. **Root Directory（根目录，最重要！）**
   - 点击 **Root Directory** 旁边的编辑按钮
   - 输入：`web`（因为前端代码在 web 文件夹里！）
   - 点击确认

2. **Branch（分支）**
   - 选择你的主分支，通常是 `main` 或 `master`

3. Railway 会自动检测到 Dockerfile，不需要配置 Build Command 和 Start Command

---

### 第 4 步：添加环境变量（最重要！）

1. 点击 **Variables** 标签
2. 点击 **New Variable** 按钮
3. 填写：
   - **Name**: `VITE_API_URL`
   - **Value**: 你的服务端地址！比如：`https://your-railway-api.railway.app`
   - 注意：地址要加 `https://`，不要有多余的空格
4. 点击 **Add** 按钮保存

---

### 第 5 步：部署！

1. 检查一下所有配置：
   - Root Directory 是 `web` ✓
   - 环境变量 `VITE_API_URL` 已经添加 ✓
   - 服务端地址填对了 ✓
2. 点击 **Deploy** 或 **Deploy Now** 按钮
3. 等待 2-5 分钟（期间可以在 **Deployments** 标签页看构建日志）
4. 部署成功后，Railway 会给你一个网址，类似：`https://ai-lowcode-platform-web.railway.app`

---

### 第 6 步：验证部署成功

1. 复制 Railway 分配给你的网址
2. 在浏览器里打开它
3. 如果看到登录页面或首页，说明前端部署成功了！
4. 试试注册/登录，看能不能正常调用 API

---

### 第 7 步：不要忘记更新后端的 CORS 配置！

1. 回到 Railway 后端服务
2. 点击后端服务 → **Variables**
3. 更新 `ALLOWED_ORIGINS`，把你的前端地址加进去：
   - 比如：`https://ai-lowcode-platform-web.railway.app`
   - 如果之前是 `*`，现在可以改成具体地址了
4. 重新部署后端服务

---

## 常见问题（按可能性排序）

### Q1: 构建失败了怎么办？

A:
1. 点击前端服务 → **Deployments** 标签
2. 点击失败的那次部署，看日志
3. 常见原因：
   - 忘记设置 Root Directory 为 `web`
   - 本地先试一下：`cd web && pnpm install && pnpm build`，看本地能不能成功
   - 检查 Dockerfile 和 nginx.conf 是不是在 web 目录下

### Q2: 页面可以打开，但所有 API 请求都失败？

A:
1. 检查环境变量 `VITE_API_URL` 有没有填对
2. 检查后端的 `ALLOWED_ORIGINS` 有没有包含你的前端域名
3. 按 F12 打开浏览器开发者工具，看 Console 和 Network 里的错误信息

### Q3: 进入某个页面后，一刷新就 404？

A:
1. 检查 nginx.conf 文件是不是在 web 目录下
2. 确认 Dockerfile 里引用了 nginx.conf
3. 重新部署一次

### Q4: 修改了环境变量，但没生效？

A:
1. 修改环境变量后，需要重新部署才能生效
2. 在 Railway 控制台的项目 **Settings** → **Environment Variables** 里修改，不要在代码里硬编码
3. 修改后，到 **Deployments** 页面，点击最新部署右边的三个点，选择 **Redeploy**

### Q5: Root Directory 找不到在哪里改？

A:
1. 在 Railway 项目页面，点击前端服务
2. 点击 **Settings** 标签
3. 往下翻，找到 **Root Directory** 或 **Service Source** 部分

## 本地测试（部署前先试试，避免浪费时间）

如果你想在本地先测试一下前端能不能跑：

```bash
# 进入 web 目录
cd web

# 安装依赖
pnpm install

# 构建
pnpm build

# 预览生产构建
pnpm preview
```

---

## 部署成功后要做的

1. 把 Railway 分配给你的网址加到简历里
2. 测试一下完整功能（登录、创建应用、知识库等）
3. 如果是为了面试展示，面试前确保服务正常，面完暂时删掉也可以省额度

---

## 修改了代码后怎么更新？

Railway 默认自动部署：
- 只要你把代码推送到 GitHub 的 main 分支，Railway 就会自动重新部署
- 你可以在 Railway 的 **Deployments** 页面看到部署进度

---

## 费用提示

- 前端是静态服务，消耗资源很少
- 你的 $5 试用额度应该能跑很久
- 记得面试前开，面完可以暂时删掉省额度
